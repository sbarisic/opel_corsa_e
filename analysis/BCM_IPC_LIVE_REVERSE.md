# BCM IPC Live-ID Reverse Engineering

Generated by `tools/bcm_ipc_live_trace.py`.

## Findings

- Real IPC capture IDs map into BCM Group A by `(extended_id >> 18) & 0x7ff`.
- Exact wire IDs often match `(table_raw & 0x1fffffff) | 0x60`, where `0x60` is the IPC/source byte seen on the direct cluster capture.
- Most matching Group A rows have direct dispatch handlers in the `0x021A40` table.
- Dispatch wrappers call a RAM getter, move the result to `r8`, then call a payload writer.
- Payload writers now identified: `0x12F06C` writes u16 big-endian, `0x12F10C` ORs bit masks, `0x12F124` writes u8, and `0x12F208` writes u32 big-endian.
- This confirms the early Group A block is a real low-speed IPC/body payload-builder region, not a table artifact.

## Practical Implication

The IPC-only frames are not enough to synthesize gauge inputs, but the BCM firmware shows how this low-speed object family is built: table row -> dispatch handler -> RAM getter -> payload writer. The next reversing step is to trace the getter RAM locations back to their writers, especially the ECU-side Group B receive handlers for `0x0C9`, `0x3D1`, `0x3E9`, `0x4C1`, and `0x4D1`.
